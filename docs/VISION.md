## §1. 한 문장 + 차별점 — 모든 결정의 기준점

전주시 시내버스 **자체 ETA 예측 모델 기반 "목표 도착형 이동계획 AI Agent"**.

기존 BIS·네이버지도·카카오맵은 **"저 버스가 지금 어디냐 / 몇 분 뒤 오냐"**까지 답한다.
우리는 한 단계 더 간다 — **"N시 M분까지 거기 도착하려면 *지금 무엇을 해야 하나*"를 역산**한다.

지방은 배차가 수십 분이라, 무턱대고 나가 기다리면 시간을 버리고, 출발 전 지도앱을 반복해 들여다봐야 했다.
중간 정류장 도착시각에 대한 **고품질 예측**이 생기면, 남들이 못 하던 이 역산이 가능해진다 —
버스에 정시성이 없고 중간 정류장 시간표 힌트가 없어 아무도 안 하던 일을, 자체 모델로 한다. **이것이 존재 이유다.**

## §2. 사용자 여정 — 역산 시나리오 (표면 목표)

사용자는 **목적지 + 목표 도착시각**만 등록한다(전날이든 임박해서든, 처음 가는 곳이든). 이후엔 알림을 따라다니기만 하면 된다:

```
[등록] 전북대 9:00 도착          ← 사용자가 하는 유일한 입력
   │
   ├─ (전날/사전) "내일 7:35쯤 나가면 됩니다"     ← 에이전트 역산 + 1차 ETA
   ├─ "5분 뒤 출발 준비"  →  "지금 나가세요"        ← 현위치 기반 역산 (완충 포함)
   ├─ "○○정류장으로 가세요" (도보)                 ← OSRM 최근접/최적 정류장
   ├─ "104번 07:48 차를 타세요"                     ← 1·2차 ETA로 선택된 배차
   ├─ (GPS가 버스 궤적 추종) "정상 탑승 확인"        ← GPS↔실황 매칭
   ├─ "다음 정류장에서 내려 환승 준비"              ← 2차 실시간 ETA
   ├─ "△△까지 도보 이동 중…"  →  "271번을 타세요"  ← OSRM + 2차 ETA
   └─ "도착". 목표시각 준수.
```

이 여정이 **두 트랙(코어·서비스)이 왜 둘 다 필요한지**를 증명한다. 어떤 작업이든 "이게 이 여정의 어디에 기여하나?"로 되돌아온다.

## §3. 시스템 큰 그림 — 2트랙 + 합류점

```
코어 트랙 ──  raw수집 ─▶ trip재구성 ─▶ 1차모델 ─▶ 2차모델 ─▶ 에이전트 ─┐
 (예측 자산)   ✅          ✅          (설계)     (예정)     (예정)    │
                                                                       ▼
                                                               ┌──────────────┐
                                                               │   serve API   │ ◀ 두 트랙이 만나는 합류점
                                                               │ (M0 더미계약✅)│   앱-facing 계약
                                                               └──────────────┘
                                                                       ▲
서비스 트랙 ─ 와이어프레임 ─▶ Flutter 앱 ─────────────────────────────┘
 (사용자 접점)  (예정)          (미착수)
```

- 두 트랙은 **병렬로** 전진하고, **serve API에서 합류**한다. (serve는 "맨 끝 단계"가 아니라 처음부터 양 트랙을 잇는 척추 — 더미계약으로 먼저 띄우고 자산 완성 계층부터 real 교체.)
- 데이터는 한 방향: `reference`(기준, 곁에 둠) │ `raw → interim → features → models → predictions`.

## §4. 통합 마일스톤 M0~M5 — 절대 좌표계

> 트랙이 합류하는 게이트. Phase(작업 단위, [ROADMAP](ROADMAP.md))와 다르다. **상태는 ✅/대기 한 단어만.**

| | 마일스톤 | 코어 트랙 조건 | 서비스 트랙 조건 | 상태 |
|---|---|---|---|---|
| **M0** | 더미계약 가동 | trip재구성 ground truth | serve 전 엔드포인트(기준데이터·실황·날씨 real, 추론·plan dummy) | ✅ |
| **M1** | 1차모델 real | 사전추론(발차→전 정류장 도착) 서빙 | pre-eta 라우터 real 교체 | 대기 |
| **M2** | 2차모델 real | 실시간추론(발차 감지 후 보정) 서빙 | live-eta 라우터 real 교체 | 대기 |
| **M3** | 에이전트 real | 목표도착 역산 + 추적·알림 | `/plan` 라우터 real + 푸시 | 대기 |
| **M4** | 앱 베타 | — | Flutter 앱(와이어프레임→구현) end-to-end | 대기 |
| **M5** | 외부 공개 | 신뢰도 검증 | 고정 URL·API키·rate limit·systemd 영속화 | 대기 |

## §5. 컴포넌트 — 한 줄 역할 + 책임 경계

> 각 컴포넌트의 **권위 정의**. "1차가 뭐였지/2차랑 뭐가 달랐지"는 여기서 즉답. 깊이는 오른쪽 문서.

| 컴포넌트 | 한 줄 역할 (입력 → 출력) | 상태 | 깊이 |
|---|---|---|---|
| **수집기** | ITS·KMA를 raw 무손실 저장 (가공 0) | ✅ | — |
| **trip 재구성** | raw → 실제 운행 trip + 구간 y라벨 (모델 무관) | ✅ | [trip-reconstruction](design/trip-reconstruction.md) |
| **1차 모델 (사전추론)** | 발차 전, 미래 버스가 **전 정류장에 도착할 clock-time** 예측. 단위 = **`(stdid, 발차슬롯)` = 한 배차**. 정적+예보+과거통계 → 구간 ETA cumsum | 설계 | [first-model](design/first-model.md) |
| **2차 모델 (실시간추론)** | 발차 **감지 후**, 1차를 prior로 실시간 위치·교통·기상 반영해 **남은 정류장 보정**. attention-transformer | 설계예정 | [second-model](design/second-model.md) |
| **에이전트** | 목적지+목표시각 → 탑승 버스열 **역산** → 현위치 기반 **지속 알림**. 완충=통계기반. 도보=OSRM | 설계예정 | [agent](design/agent.md) |
| **serve API** | 두 트랙 합류점. 앱-facing REST 계약 (계층: 기준데이터·실황·사전추론·실시간추론·plan·weather) | 부분real (M0) | [serve-api](design/serve-api.md) · [app-api-flow](design/app-api-flow.md) |
| **앱 (Flutter)** | 사용자 입출력 + 알림 수신 (iOS/Android) | 미착수 | [wireframe](design/wireframe.md) |

## §7. 불변 원칙 — 요지 (전문 → [CLAUDE.md](../CLAUDE.md))

- **잘게 쪼개고, 작업마다 commit + STATUS 갱신.** 둘 다 길을 잃지 않게.
- **임시방편 금지, 품질 우선.** 시간이 없으면 범위를 줄이되 품질은 안 낮춘다.
- **큰 결정은 먼저 상의.** 구조 변경·배포·삭제/덮어쓰기·팀 리소스 영향.
- **추측 말고 측정·검증으로 결정.** 원본 맹신 금지, 성능은 실측.
- **작년 코드는 정답 아님.** 아이디어만 가져오고 뿌리부터 재설계.
