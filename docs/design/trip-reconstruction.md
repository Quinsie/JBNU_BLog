# Trip 재구성 + 출발 검출 (Phase 3) — 데이터 파악 기반 설계

> raw busPosList → 차량별 trip 시퀀스 → **실제 출발/도착 검출 + 노이즈 필터** → 구간 ground truth.
> *모델 무관* 단계지만, **1차 모델 y 라벨을 정의**한다(앵커 = 실제 출발 검출 시점; [first-model.md](first-model.md) §3.1).
> **상태**: v1.1 + 보강 8개 항목 완료. 구현 `src/preprocess/trip_reconstruct.py`, 출력 스키마 [DATA_SCHEMA](../DATA_SCHEMA.md) `interim/trips/`.
> §2 의 초기 통계는 5/26 단발 20분 수집 기준(설계 출발점). **확정된 풀데 funnel·검증결과는 [DATA_NOTES](../DATA_NOTES.md) 5/27 항목** (시간표 슬롯의 91% 학습 trip 으로 살림).

> ⚠️ **이 휴리스틱은 한 번에 확정하는 게 아니라, 데이터 보며 지속적으로 고치는 반복개선 대상**이다(임시방편 아님 — §3·§4의 예외 케이스가 계속 발견될 것).

## 1. raw 레코드 구조 (실측 확인)
한 줄(폴링 1회): `{ts, stdid, status, ok, body:{busPosList:[...]}, elapsed_ms}`.
`busPosList[i]` = 그 노선에 *그 순간 떠있는 버스 1대*:
- `PLATE_NO` — 차량번호판. **trip 식별 키**(작년 lock-on 대체). stdid(=노선·방향) 내에서 한 plate 시퀀스 = 한 trip.
- `LATEST_STOP_ORD` — 마지막 통과 정류장 순번(진행도).
- `CURRENT_NODE_ORD` — 노드 순번. `node_ord − stop_ord` 오프셋은 한 trip 내에서 **0→5처럼 단조 증가**(특정 구간에 중간노드가 더 끼며 누적). 노선에 따라 상수 오프셋(예 +5 고정)이기도, 규칙적 증가이기도 함. 깨진 값 아님. 단 위치신호는 `LAT/LNG`가 훨씬 직접적이라 **node_ord는 1차·2차 모두 미사용(폐기)**.
- `LAT/LNG`, `SPEED`, `LOW_FLAG`(저상), `COMPANY_NAME`, `TELNO`.
- ⚠️ **이 9개가 전부** — 발차/도착/회차/순번 필드는 `selectBisRouteLocationList` 에 **없다**(실측 확인). 실제 발차의 직접 소스는 GPS 궤적뿐.

## 2. 실측 통계 (2h, 446노선)

- trip 조각 366개: **ORD 1부터 포착 24%** / 중간진입 76% / 종점도달 44%.
  - ⚠️ 24%는 *2시간 단발 수집의 artifact* — 상시수집 시 출발 포착률 급상승(못 쓰는 건 수집 시작 순간 진행 중이던 trip뿐).
- **시작ORD 체류**: median 1.9분 / p90 10.2 / max 17.1 (출발 전 대기, 롱테일=대기노이즈).
- **종점 체류**: median 0.8분 / p90 11.0 / max 19.2 (도착 후 layover, 롱테일=추적기 미종료).
- **실측 배차간격**: median 11.9분(노선별 연속 출발 차). 시간표 간격과 부합.

### 2.5 저녁 상시수집 실측 (2026-05-26 20:18~24:00, `.74` 우회분)
- **풀 trip 다수 포착**: 기점(ord=1)부터 잡힌 trip ≈168개 — 상시수집 효과 입증(20분 단발의 24%와 대비).
- **정차 중 GPS 거동**(저속 spd≤5, n≈9,825): 틱간 변위 **정확히 0이 67.9%**(좌표 비트-held), ≤10m가 80.5%. → 정차 시 작은 연속지터는 드묾(API가 좌표 held).
  - 단 **꼬리가 위험**: p99=293m, max=10.5km. = GPS 글리치 + speed/좌표 비동기 stale. → **단일 틱 변위로 발차 판정 금지**(글리치 오검출).
- **GPS 갱신율 가변**: 10초마다 실제 변동하는 버스도, 20~30초 held 후 점프하는 버스도 있음. → 10초 폴링은 **오버샘플 아님**(빠른 갱신 버스를 위해 필요). held 행은 느린 갱신 버스의 부산물.
- **stop_ord 의미 재확인**: `ord=N` = "정류장 N 통과, N+1 미도달" = **N→N+1 주행 중**. `ord`가 N+1로 바뀌는 건 N+1 *도착* 순간. → 발차는 `ord` 전이로 못 잡음(ord=1 run 안에 발차대기+1구간주행이 섞임).
- **분리 독립성**: 재구성은 수집기와 무관한 순수 후처리 → 쌓이는 raw로 **오프라인 리플레이 반복시험** 가능(수집 영향 0).

## 3. 관측된 노이즈 유형 (실데이터에서 확인)
- **중간진입**: 수집/운행 시작 전 이미 출발 → 낮은 ORD 못 봄. (예: PLATE 2048 ORD 18부터)
- **출발 전 장기 대기**: 시작ORD에서 수~17분 정지 후 출발. → 시간표/첫관측 앵커가 깨지는 지점.
- **종점 미도달 정지**: 종점 직전 ORD에서 멈춤(예: 1757이 47에서 6.8분). 저속이동 vs 정지 구분 필요.
- **종점 후 추적기 방치**: 종점 ORD에서 장기 체류 후 소멸(예: 1719가 48에서 9분).
- (예상) ORD 역행(GPS 지터), 관측 공백(이번 창에선 최대 6s로 양호).

## 4. 출발/도착 검출 휴리스틱 (설계 — 실데이터 검증 반영)

**철학: 완벽한 trip 경계를 목표하지 않는다. *믿을 수 있는 구간*을 목표한다.** y가 구간 단위라
경계 오류는 첫/마지막 구간만 건드리고 중간 구간은 멀쩡 → 엣지케이스는 데이터를 오염시키지 않고
*몇 구간을 못 쓰게* 할 뿐(우아한 강등). 그래서 모든 케이스를 if문으로 잡지 않고,
**진입/소멸점만 제대로 잡고 + 지나치게 흔들리는 것만 폐기**한다(A/B/C 다단계 등급은 과폐기라 기각).

### 4.0 trip 분할 — *자정 무경계(연속성 기반)*
**trip 은 달력 날짜가 아니라 연속성으로 정의.** load 는 인접일(전·당·다음날)을 하나의 연속 스트림으로 병합하고, 분할은 **gap·ord리셋(터미널 회차) 신호로만** 한다. trip 은 "시작(첫 관측) 날짜"가 소유(`owner != 대상일`이면 인접일 처리에서 잡으므로 스킵 → 중복 0). 효과: ① 23h→00h crosser 온전 ② **24시간 운행버스(수도권 등 전국확장 대비)도 회차마다 분할**되어 시각 무관 ③ service-day 고정경계(04:00 등) 불필요(급한불끄기 회피).
- stdid 내 `PLATE_NO`로 그룹 → 시간순. 동일 plate가 한참 뒤 **낮은 ord로 재등장 = 다음 회차** → 분할.

### 4.1 발차 검출 — *시간표 prior로 앵커한 GPS 지속 이탈*
`ord` 전이는 발차가 아님(다음 정류장 도착시점). 발차는 시작(최소) ord run **안에서** 찾는다:
1. **기점 정차상태**: 선행 틱들이 안정 중심점(centroid) 반경 `R_park`(~30m) 안에 머무는 구간(좌표 held 허용).
2. **발차 = 지속 이탈**: 중심점에서 `R_depart`(~50m) 넘어 멀어지고 **이후 K틱(~30s) 동안 되돌아오지 않고 계속 멀어짐**. → 한 틱 튀었다 복귀하는 글리치 기각, *단조 이탈*만 인정.
3. **발차시각** = 그 지속 이탈 시작 틱(정차 마지막 틱). `SPEED` 상승은 **보조 확증**(단독 트리거 아님).
4. **시간표 prior**: `timeList`(예정 발차) 슬롯 근접이면 발차로 강하게 인정, 슬롯과 동떨어진 미세이동은 지터로 강등 → `R` 임계의 절대 애매함을 prior가 보정.
5. **포착 품질 분기**: 정차상태 못 봄(첫 관측부터 이동)=`origin_moving`(발차 근사/플래그) / 시작 ord>1=`mid_entry`(발차 없음, 구간라벨만).

### 4.2 종료 검출 — "종점 도달"이 아니라 "진행 정지 + 이후 미진행/소멸"로 일반화
종점 도달을 종료로 가정하면 안 됨(조기회차·대기 존재). 아래 케이스를 한 규칙으로 흡수:

| 케이스 | 처리 |
|---|---|
| 종점 도달 후 소멸 | 정상 종료. 마지막 ord run 시작 = 종점 도착 |
| 종점 도달했는데 안 사라짐(추적기 방치) | ord 진행 정지 + 이후 미진행 → 그 시점 종료, 잔여 체류 버림 |
| 중간에서 갑자기 소멸(관측 공백) | 마지막 관측까지를 trip으로, `disappeared_mid` 플래그(불완전=부분사용) |
| 종점 전 조기 회차 | ord 최대점<종점 → `early_turnaround`, 도달 구간까지만 |

### 4.3 글리치/노이즈 필터 (전역)
- **글리치 제거**: 틱간 추정속도가 비현실값(예 >120km/h)인 좌표점프, 단발 outlier 좌표 제거(발차뿐 아니라 모든 위치처리에 적용).
- **폐기 기준**: 역행 과다·대형 관측공백·글리치 범벅 trip은 통째 폐기. 그 외엔 keep(부분구간 활용).

### 4.4 시간표 매칭 (라벨 + QA) — 노선전역 1:1 배정
검출 발차열 ↔ 예정슬롯열을 **노선전역 시간순보존 1:1 배정**(단조 매칭 DP, `(매칭수 최대, 총|delta| 최소)` 사전식, 슬롯당 1회, 게이트 `MATCH_GATE_SEC`=600s 내, 순서 교차 불가)으로 매칭 → *"몇시 배차"* 라벨(serving 열거·시간대 feature용. **배차간격은 feature 아님**). 동시에 §4.6 QA 입력.
- **greedy 최근접을 버린 이유**: 트립별 독립 최근접은 ⓐ 두 버스가 같은 슬롯에 중복매칭(검증: 5/26 4노선), ⓑ 수집중단으로 발차타이밍 밀리면 엉뚱한 슬롯에 붙음, ⓒ 벽지 슬롯공백에서 8787s급 무의미 매칭. 전역배정이 ⓐⓑ를 구조적으로 제거하고 게이트가 ⓒ를 차단(max delta 8787s→584s).
- **on_schedule**: 배정됨=true / 발차검출O 미배정=false(추가운행·검출이상) / 발차없음=null.
- **미매칭 슬롯**(배정 안 된 예정발차)은 trip 밖 `_match_diag.jsonl` 로 노선단위 리포트 → 수집공백·미운행·검출누락 가시화(부분수집창이면 대부분 미매칭 = 정상).

### 4.5 작년 방식 대비 (피해야 할 것)
- 작년 `trackSingleBus.py`: 락온 시 **ORD 1 시각을 시간표값으로 하드 고정** + ORD 2 보간 → *실제 출발 오염*. 추적은 `CURRENT_NODE_ORD`+route_node 매칭(반면 추론은 STOP_ORD = 단위 불일치).
- 작년은 *증상 땜질*(보간·종점휴리스틱·vtx수작업)은 했으나 *뿌리*(node/stop 혼용·출발 강제)는 미해결. → 이번엔 **실제출발 검출 + STOP_ORD 일관 + 시간표는 QA/매칭만**.

### 4.6 시간표 교차검증 (데이터 품질 QA)
검출 출발 vs 시간표 출발 비교:
- **일치** → 수집·해석·시간표 모두 정상 확인.
- **불일치** → 원인 셋: ① 우리 검출/해석 오류 ② 시간표 미갱신(변경 반영 안 됨) ③ 실제 off-schedule(증/결행·임시).
- **집계 패턴으로 원인 분리**: 한 노선 *체계적* 불일치=시간표 미갱신 / *산발적*=검출노이즈 or 개별 off-schedule.

### 4.7 LATEST_STOP_ORD 얼음 → GPS 근접매칭 복원 (API 불신·교차검증)
**원칙: API 를 신뢰하지 않고 독립 신호(GPS+정적 정류장좌표)로 검증·보정.** 일부 차량(장거리·시외성)에서 `LATEST_STOP_ORD` 가 수분간 얼었다가 여러 정류장을 건너뛰어 점프(원천 telemetry 갱신주기가 폴링보다 느림). GPS 로 검증 시 버스는 그 정류장들을 물리적으로 통과했음. 처리(두 경우):
- **GPS 는 움직이는데 ord 만 얼음** → 점프 gap 시간창에서 정류장좌표에 `R_STOP_MATCH`(150m, 결측 ord별 GPS최근접 분포 p60≈160m 에서 결정) 안으로 든 GPS 샘플로 통과시각 **복원**(`src="gps"`).
- **GPS 도 같이 얼어 근접 못 함** → 복원불가 = **쓸 수 없는 구간** → 비움(`stops_unrecoverable`), 가짜로 안 채움.
- **보간 금지**(작년 오염방식): 실제 GPS 샘플만 사용. 5/26 전노선 결측 ord 862 중 57% 복원, 구간의 1.9%가 gps복원분(`src` 태그로 학습 시 신뢰도 선택).

## 5. 1차 모델 라벨 수율과의 연결 (핵심)
- y = **구간(정류장간) 소요시간** (`ord=N` run 지속 ≈ N→N+1 주행). **누적(cumsum)은 학습 타깃 아님 — 추론 후 도착시각 복원 시에만** 합산(first-model §3.1).
- 그래서 **중간진입 trip도 interior 구간은 그대로 유효**(예: 18→31 전 구간). 발차 앵커가 *반드시* 필요한 건 **첫 구간(출발→1번)과 누적 복원**뿐 → trip당 손실은 최대 첫 구간 1개.
- → 결정1(출발검출)·결정2(구간타깃)가 상호 보완. "시점 데이터가 많이 버려진다"는 우려를 구간타깃이 흡수(우아한 강등).
- **구간 신뢰도**: 각 segment `src`(ord/gps)로 분리 → 1차 학습 시 ord-only(고신뢰)만 쓰거나 gps복원분을 가중·포함 선택 가능(§4.7).
